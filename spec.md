# Margli — Spec: режим «Обзор магазина» (фаза 2)

Статус: одобрено (владелец «продолжай»), транспортно-независимый фундамент реализован 2026-06-19 (7 lib-модулей, 317 тестов зелёных). Осталось: UI таблицы + live-recon листинг-транспорта. Дата: 2026-06-19. Проект: margli-extension.
Фаза 2 поверх текущего MVP (одна карточка + myShopId). MVP не ломаем.

---

## 1. ЦЕЛЬ

Дать селлеру Kaspi за один ввод (ссылка или название его магазина) таблицу по
ВСЕМ его товарам сразу: цена, комиссия Kaspi 2026, налоги РК, есть ли демперы и
на сколько, и сколько остаётся до закупки. Себестоимость селлер дозаполняет
(вручную или CSV) и тогда видит реальную чистую прибыль по каждому SKU и по
магазину целиком. Принцип: всё, что можно взять из Kaspi, берём автоматически;
то, чего в Kaspi нет (закупка, карго, реклама), не выдумываем, а просим у селлера.

---

## 2. ЧАСТИ СИСТЕМЫ

1. **Резолвер магазина** — добывает `merchantId` БЕЗ ручной ссылки (у селлера её
   обычно нет под рукой): (а) если открыта витрина магазина `shop/m/<id>` — из
   URL; (б) если селлер на карточке своего товара и задан myShopId — `merchantId`
   из ссылки его магазина в блоке продавцов (плагин это уже умеет в drawer,
   inferShopId из href `/shop/m/<id>`); (в) ручной ввод ID — запасной вариант.
2. **Парсер листинга магазина** — по `merchantId` обходит страницы магазина и
   собирает все товары (название, SKU, цена, ссылка, категория). Дёшево: 1 запрос
   на страницу (~12-24 товара), не на товар.
3. **Демпинг-движок** — для конкретного SKU зовёт offers-API (`kaspi-offers-api.ts`),
   возвращает мин. конкурента и число демперов. Дорого: 1 запрос на товар.
   Работает по очереди с throttle и приоритетом (см. раздел 5).
4. **Расчётный слой** — на каждую цену считает комиссию Kaspi (`kaspi-fees.ts`),
   налоги РК (`kz-taxes.ts`) и, если есть себестоимость, чистую прибыль
   (`margin-calc.ts`). По выбранной орг-правовой форме (ИП/ТОО + режим)
   автоматически подставляет применимые ставки и отчисления (НДС и порог, ОПВ/СО/
   ВОСМС, соцналог) — селлер не считает руками. Каждая ставка помечена датой и
   первоисточником (см. раздел «Актуальность»). Чистые функции, без сети.
5. **Себестоимость-store + CSV-импорт** — хранит cost-профиль по SKU (закупка,
   доставка, реклама, возвраты, категория). CSV из кабинета Kaspi мапится в эти поля.
6. **UI «Обзор магазина»** — виртуализированная таблица (рендер только видимых
   строк), сортировка/фильтр, прогресс-бар загрузки, кнопка «обновить».
7. **Кэш** (`chrome.storage.local`) — товары + цены + демпинг-результаты по
   `merchantId` с метками времени. Инкрементальное обновление, не парсим заново
   каждый раз.

## 3. ВХОД → ВЫХОД каждой части

| Часть | Вход | Выход | С кем говорит |
|---|---|---|---|
| Резолвер магазина | контекст (URL витрины / карточка товара + myShopId) или ручной ID | `merchantId` или ошибка | UI, drawer, страница магазина |
| Парсер листинга | `merchantId`, номер страницы | массив `{sku, name, price, url, category}` + есть ли ещё страницы | Kaspi (страница магазина), Кэш |
| Демпинг-движок | `sku`, `cityId` | `{minCompetitor, dumpersCount, competitors[]}` | Kaspi offer-view API, Кэш |
| Расчётный слой | `{price, categoryId, cost?...}` | `{kaspiFee, tax, netProfit?, marginPct?}` | никем (чистые функции) |
| Себестоимость-store | `sku` или CSV-файл | cost-профиль(и) | chrome.storage, UI |
| UI | данные из кэша/движков | таблица + прогресс | все части |
| Кэш | `merchantId` | сохранённый снимок + timestamp | storage |

Колонки таблицы (одна строка = один SKU):
товар · цена · комиссия Kaspi · налог · **остаток до закупки** (цена минус
комиссия минус налог) · мин. конкурента · демперов · [себестоимость, ввод] ·
**чистая прибыль** (только если себестоимость задана, иначе «—»).

---

## 4. ДАННЫЕ

Хранилище: `chrome.storage.local` (как и весь MVP). Новые ключи:

- `margli:store:<merchantId>` — снимок магазина: `{merchantId, name, fetchedAt,
  products: [{sku, name, price, url, category}], dumping: {sku: {min, dumpers, at}}}`.
- `margli:costs` — уже есть в MVP (cost-профили по SKU), переиспользуем.
- `margli:store:progress` — прогресс текущей загрузки (страниц/демпинга), чтобы
  показывать бар и возобновлять.

Источники: страница магазина Kaspi (DOM/листинг) для товаров и цен; offer-view
API для демпинга; ввод селлера/CSV для себестоимости. Поиск по точному SKU,
не семантический.

## 5. МАСШТАБ — как быть с большим магазином (главный раздел, ответ на вопрос владельца)

Проблема: у крупного селлера сотни-тысячи SKU. Наивно «дёрнуть демпинг по
каждому» = сотни-тысячи запросов = бан Kaspi и зависший UI. Разделяем дёшево и дорого:

**Цены (дёшево):** обход листинга магазина постранично, 1 запрос на страницу.
1000 товаров ≈ 50-80 страниц. С паузой ~0.5-1с между страницами это 1-2 минуты,
с прогресс-баром «загружено X из Y товаров». Цены всех товаров получаем всегда.

**Демпинг (дорого, 1 запрос на товар):** НЕ запускаем на все сразу. Стратегия:
1. **Приоритет:** сначала товары под наблюдением (watchlist) и те, где селлер
   уже задал себестоимость (значит они ему важны), потом самые дорогие.
2. **По требованию:** демпинг для строки считается, когда селлер раскрывает её
   или прокручивает к ней (lazy on-scroll), а не для невидимых 950 строк.
3. **Фоновая очередь с throttle:** не более 1-2 запросов параллельно, пауза
   ~1-2с (с небольшим джиттером), бэкофф при 429. Можно поставить на паузу.
4. **Лимит по умолчанию:** авто-демпинг для топ-N (например 50). Остальное по
   кнопке «досчитать демпинг по всем» с явным предупреждением, что это долго.
5. **Кэш демпинга:** результат живёт с TTL (например 1-3 часа); повторный заход
   не перезапрашивает свежее.

**UI:** виртуализация таблицы (react-window или tanstack-virtual) — в DOM только
видимые ~20-30 строк, иначе 1000 строк вешают вкладку.

**Анти-бан:** последовательная очередь, джиттер задержки, бэкофф на 429/403,
дневной потолок запросов. Если Kaspi начал отвечать ошибками — стоп и сообщение
селлеру, а не долбёжка.

**Граница честности:** для очень больших магазинов (1000+ SKU) плагин тянет с
throttle, но это медленно. Идеальное место для такого объёма — сервер, который
парсит и кэширует централизованно. Это явная не-цель текущей фазы (см. раздел 7).

## 5b. Актуальность ставок и орг-форма (требование владельца 2026-06-19)

Налоги РК и правила Kaspi МЕНЯЮТСЯ (НДС 12→16% с 2026, новая шкала комиссий Kaspi
с 5.01.2026). Захардкодить и забыть нельзя. Требования:

- **Орг-форма → автоналоги:** селлер выбирает форму (ИП упрощёнка / ИП ОУР / ТОО
  упрощёнка / ТОО ОУР / розничный) → плагин сам проставляет применимые ставки:
  ИПН/КПН, НДС и порог регистрации, ОПВ/СО/ВОСМС/ОПВР, соцналог. Селлер не помнит
  цифры. Он часто меняет форму/магазин — смена пересчитывает всё.
- **Каждая ставка версионирована:** значение + «действует с» + «проверено (дата)»
  + ссылка на первоисточник (adilet.zan.kz / kgd.gov.kz / guide.kaspi.kz).
  Частично уже в `kz-taxes.ts` / `kaspi-fees.ts`, довести до всех ставок.
- **Предупреждение об устаревании:** если текущий год > года ставок или ставки
  старше N месяцев — мягкий баннер «проверьте актуальность, ставки на 2026,
  источник [ссылка]». Не блокирует, но честно предупреждает.
- **Кнопка «сверить с источником»** рядом со ставкой → открывает первоисточник.
- **Регресс-тест:** источники первичны и живы (как `FORBIDDEN_DOMAINS`-гард уже
  в `kz-taxes.test.ts`, который блокирует банковские/блогерские ссылки).

**Граница честности:** плагин НЕ следит за изменениями закона/Kaspi в реальном
времени сам — для этого нужен сервер или живой фид (фаза 3) либо ручное обновление
ставок по сессии. Плагин гарантирует прозрачность (дата + источник + предупреждение
об устаревании), а не магическую авто-актуальность. Не делаем вид, что цифры
обновляются сами.

## 6. СЦЕНАРИИ (сквозные)

1. **Малый магазин (20 SKU):** селлер вставил ссылку своего магазина → за
   несколько секунд таблица из 20 товаров с ценами, комиссиями, налогами;
   демпинг досчитался по всем (20 запросов с throttle); селлер видит, что по 3
   товарам его демпингуют. Себестоимость пустая → колонка прибыли «—».
2. **Дозаполнение себестоимости:** селлер скачал CSV из кабинета Kaspi, перетащил
   в плагин → cost-профили подтянулись по SKU → колонка «чистая прибыль»
   заполнилась, внизу итог по магазину «реальная прибыль за месяц ≈ X».
3. **Крупный магазин (800 SKU):** цены всех 800 загрузились за ~1.5 мин с
   прогресс-баром; демпинг авто-посчитался по топ-50 + watchlist; для остальных
   селлер видит «демпинг по запросу», может нажать «досчитать всё» (предупреждение
   о времени) или просто скроллить, и демпинг догружается по видимым строкам.
4. **Повторный заход:** селлер открыл магазин снова через час → данные из кэша
   мгновенно, кнопка «обновить» подтягивает изменения инкрементально.

## 7. НЕ-ЦЕЛИ (в этой фазе НЕ делаем)

- Сервер/бэкенд для парсинга (всё в браузере селлера; сервер это фаза 3).
- Несколько магазинов одновременно в одной таблице (один merchantId за раз).
- Мульти-маркетплейс (WB/Ozon) — вердикт исследования против.
- Авто-демпинг по всем тысячам SKU без явного запроса (анти-бан).
- Реал-тайм мониторинг цен магазина в фоне (это watchlist MVP, отдельно).
- Угадывание себестоимости/прибыли без данных селлера (принцип честности).
- Резолв названия → магазин со 100% точностью, если Kaspi не даёт поиска по
  магазинам (тогда вход = ссылка магазина, см. открытые вопросы).

---

## 8. ДОСТУПЫ ОТ ЧЕЛОВЕКА

- **Ничего вводить руками не обязательно:** merchantId плагин берёт сам, когда
  селлер открыл свой товар (при заданном myShopId) или свою витрину `shop/m/`.
  Ручной ввод ID — только запасной вариант.
- **CSV из кабинета Kaspi** для себестоимости — селлер выгружает сам (у плагина
  нет доступа к кабинету `/mc/*`, он отключён в MVP).
- Ничего платного/секретного: API-ключей нет, всё через публичные страницы и
  уже-залогиненную сессию селлера.

---

## 9. ГОТОВО-КОГДА (передаётся в /goal)

- По ссылке магазина таблица всех товаров с ценами, комиссией, налогом и
  «остатком до закупки» строится автоматически.
- Демпинг считается по приоритету/по требованию с throttle и прогрессом, без
  бана на тестовом магазине среднего размера.
- CSV-импорт себестоимости заполняет колонку реальной прибыли; итог по магазину
  считается.
- Таблица не вешает вкладку на 500+ строк (виртуализация).
- Кэш: повторный заход мгновенный, «обновить» работает инкрементально.
- Юнит-тесты на резолвер, парсер листинга (фикстуры), throttle-очередь, маппинг
  CSV, расчётный слой; покрытие как в MVP (все зелёные). Live-smoke на реальном
  магазине через Claude-in-Chrome MCP перед «готово».

## Открытые вопросы / риски (подтвердить ДО кода)

1. **Добыча merchantId без ссылки** (у владельца ссылки нет под рукой). Основной
   путь: merchantId из ссылки магазина селлера в drawer (когда он на своём товаре,
   myShopId совпал) или из URL витрины `shop/m/`. Проверить живьём: стабилен ли
   этот merchantId и ведёт ли на полный листинг товаров магазина. Поиск по
   названию — только если Kaspi реально его даёт (скорее нет).
   **РАЗВЕДАНО 19.06:** на карточке товара название продавца жёстко привязано к
   merchantId (hoco.→30386321, QPick→16033005, Astana-case→Astanacase; бывает
   числовой И буквенный-slug). Рабочий UX: селлер вписал название → один раз
   открыл любой свой товар → плагин поймал merchantId по совпадению названия →
   запомнил → дальше «Обзор магазина» тянет все товары авто. Чистого
   name→merchantId без открытия товара у Kaspi нет.
2. **Листинг магазина — РАЗВЕДАНО 19.06.** Витрина мерчанта живёт на SPA-странице
   `kaspi.kz/shop/info/merchant/<merchantId>/` (заголовок «<магазин> — товары…»).
   Прямой fetch/navigate отдаёт **404** (client-side routing) — поэтому ссылка
   `/shop/m/<id>` из DOM напрямую не открывается, только клик внутри SPA ведёт на
   `/shop/info/merchant/<id>/...`. Полный список товаров мерчанта (все SKU с
   пагинацией) тянем парсингом merchant-страницы ИЛИ через внутренний `/yml/` API
   (найти через network-перехват при кодинге). Простого offer-view-эквивалента для
   товаров мерчанта пока не нашли — это первый шаг при реализации.
3. **Реальный лимит запросов Kaspi** до 429/бана — подобрать throttle опытно на
   среднем магазине, начать консервативно (1 запрос / 1.5-2с).
4. **CSV-формат кабинета Kaspi** — какие колонки реально даёт экспорт, как мапить
   в cost-профиль. Нужен пример файла от селлера.
5. **Где живёт UI обзора** — отдельная вкладка в popup (тесно) или полноэкранная
   страница расширения (`chrome.tabs.create` на свою страницу). Для широкой
   таблицы скорее полноэкранная страница. [предположение: отдельная страница]
